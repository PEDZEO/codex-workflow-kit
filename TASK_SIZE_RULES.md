# TASK_SIZE_RULES.md

Размер задачи должен влиять на процесс. Не используй одинаковый workflow для мелкого фикса и большого изменения.

## Small

Примеры:
- точечный багфикс;
- правка одного обработчика;
- небольшое изменение текста или условия.

Правила:
- держать работу локально;
- без агентов, если это не экономит время прямо сейчас;
- минимальный релевантный тест;
- краткое обновление памяти.

## Medium

Примеры:
- несколько файлов в одной зоне;
- изменение handler + keyboard + test;
- локальный рефактор без изменения архитектуры.

Правила:
- короткий план обязателен;
- можно делегировать поиск тестов или review;
- ownership файлов должен быть явным;
- приемка и targeted verification обязательны.

## Large

Примеры:
- изменение нескольких зон;
- изменение контракта или схемы;
- добавление нового workflow.

Правила:
- сначала план;
- декомпозиция по зонам;
- делегирование только по непересекающимся областям;
- отдельная проверка рисков;
- память обновлять аккуратно, без длинной истории.

## Cross-Cutting

Примеры:
- auth;
- config migration;
- event/API contract change;
- shared library update.

Правила:
- критический путь держать локально;
- агентов использовать только на боковые, четко ограниченные подзадачи;
- проверять `RISK_PATTERNS.md` обязательно;
- проверять контракты и связанные тесты обязательно.
