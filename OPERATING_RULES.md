# OPERATING_RULES.md

Это жесткие правила. Их задача не рекомендовать, а ограничивать поведение Codex так, чтобы он не терял контекст и не расползался по задаче.

## Rule 1: One Source Of Truth

Для текущей задачи существует только один живой источник правды:

- `.codex/memory/current-task.md`

Нельзя держать параллельные "текущие статусы" в:
- длинных сообщениях;
- временных заметках;
- handoff-описаниях;
- итоговых отчетах.

Если текущее состояние задачи изменилось, сначала обнови `current-task.md`, потом продолжай работу.

## Rule 2: Mandatory Start Sequence

Перед любой заметной работой обязательно:

1. прочитать `CODEBASE_INDEX.md`;
2. прочитать `TASK_INTAKE.md`;
3. прочитать `.codex/memory/current-task.md`;
4. только потом искать по коду.

Нельзя начинать с широкого чтения дерева, если эти файлы существуют и не признаны устаревшими.

Если `CODEBASE_INDEX.md`, `TEST_INDEX.md` или `CHANGE_AREAS.md` отсутствуют или явно устарели, сначала используй скрипты:

1. `tools/scan_project.py`
2. `tools/init_index.py`
3. `tools/init_test_index.py`
4. `tools/init_change_areas.py`
5. `tools/explain_scan.py`

Ручное заполнение без попытки structural scan должно быть исключением, а не стандартным путем.

## Rule 3: Search Must Be Narrow

Нельзя начинать поиск с хаотичного чтения файлов.

Обязательный порядок:

1. `FILE_PRIORITY.md`
2. `SEARCH_PLAYBOOK.md`
3. узкий `rg --files`
4. узкий `rg`
5. точечное чтение

Если найден рабочий flow, поиск должен остановиться.

## Rule 4: Research Must Collapse Into Summary

После любого этапа исследования результат должен быть сжат в короткое summary в `current-task.md`.

Нельзя тащить вперед:
- длинные логи;
- длинные цитаты кода;
- повторные объяснения уже подтвержденного.

Если summary не сделан, исследование считается незавершенным.

## Rule 5: Delegation Requires Ownership

Нельзя делегировать задачу без:

- явной цели;
- явного списка owned files;
- явного out-of-scope;
- критерия готовности;
- записи в `.codex/memory/handoffs.md`.

Если ownership пересекается, задача должна быть разбита заново или оставлена локально.

## Rule 6: Keep Critical Path Local

То, что блокирует следующий шаг, должно оставаться у главного Codex.

Агентам отдавать только:
- side work;
- narrow implementation slices;
- review;
- targeted verification.

Нельзя отдавать агенту неопределенную центральную задачу и ждать, что он сам построит правильный план.

## Rule 7: Every Change Needs Acceptance

Нельзя завершать задачу до прохождения `ACCEPTANCE_GATES.md`.

Минимум обязательно:
- scope;
- behavior;
- verification;
- regression check;
- memory update.

## Rule 8: Memory Must Stay Small

`current-task.md` должен быть коротким рабочим файлом.

Жесткие ограничения:
- только текущая задача;
- только актуальное состояние;
- только нужные next steps;
- без длинной истории.

Если файл разрастается, его нужно ужать, а не продолжать дописывать.

## Rule 9: Final Output Must Match Verified Reality

В финальном ответе можно утверждать только то, что:
- реально изменено;
- реально проверено;
- или явно отмечено как непроверенное.

Нельзя маскировать отсутствие проверки общими формулировками.

## Rule 10: If The System Is Not Updated, It Will Drift

Если изменились:
- структура проекта;
- карта тестов;
- зоны изменений;
- базовые entrypoints;

то соответствующие index-файлы нужно обновить до или сразу после завершения задачи.

Иначе система памяти и поиска быстро перестанет работать.

## Rule 11: Prefer Scripted Regeneration Over Manual Rebuild

Если есть локальные workflow-скрипты, сначала используй их для:

- генерации draft index;
- генерации draft test map;
- генерации draft change areas;
- объяснения вывода scanner;
- частичного refresh существующего индекса.

Полная ручная перепись этих файлов без причины создает drift и делает workflow менее повторяемым.
